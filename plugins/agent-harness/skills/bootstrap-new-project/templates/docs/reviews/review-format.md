# Формат вердикта проверяющего

На этапе **VERIFY** проверяющий (роль агента) сверяет сделанное с требованиями задачи и с `CODE-REVIEW.md`, затем выдаёт вердикт в этом формате. Из вердикта берутся числа для статистики (`docs/analytics/runs.md`).

## Что проверяется
1. **Требования задачи** — каждый пункт «что должно получиться»: выполнено / частично / не выполнено.
2. **Грабли** — не наступили ли на известные ошибки (из задачи и из `CODE-REVIEW.md`).
3. **Чек-лист проекта** (`CODE-REVIEW.md`) — каждый пункт: выполнено / не выполнено / не относится.
4. **Тесты** — есть ли тесты на новую функциональность.
5. **Безопасность** — обработан ли ввод, нет ли утечек.

## Серьёзность замечаний
- **Критично** — блокирует выпуск (нарушение требования или критичного правила).
- **Важно** — желательно исправить до выпуска.
- **Мелочь** — на усмотрение.

## Итог вердикта (шаблон)

```
Вердикт: ОДОБРЕНО | НУЖНЫ ПРАВКИ
Сводка: <1–2 предложения — что проверял и общий итог>

Замечания: критичных N · важных N · мелких N
- [критично] <файл:строка> — <что не так> → <как исправить>
- [важно]    <файл:строка> — <…>

Требования:
- [выполнено]      <пункт> — <где в коде>
- [не выполнено]   <пункт> — <чего не хватает>

Чек-лист проекта (CODE-REVIEW.md):
- [выполнено]     <правило>
- [не относится]  <правило> — <почему не применимо>
```

**Правило:** «ОДОБРЕНО» — только если нет ни одного критичного и ни одного «не выполнено». Иначе «НУЖНЫ ПРАВКИ».

## Куда идут числа
После вердикта агент дописывает строку в `docs/analytics/runs.md`: статус (одобрено/правки) и числа замечаний (критичных/важных/мелких). Подробности — `docs/analytics/README.md`.
