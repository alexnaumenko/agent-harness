# ADR-0001: Принять базу знаний и обвязку агента

**Дата:** {{DATE}}
**Статус:** Accepted
**Источник:** развёрнуто через skill `/bootstrap-new-project` (методология insight-bot-mvp + статья vc.ru/ai/2869178).

## Контекст
Новому проекту нужна (а) долговременная память, переживающая между сессиями Claude, и (б) дисциплина процесса разработки от требований до production со встроенной статистикой.

## Решение
Принять **двухслойную базу знаний + обвязку**:
- **`wiki/`** — структура (производное от кода: схема БД, маршруты, архитектура).
- **`docs/`** — процесс (сессии, ADR, audits, runbook, статистика).
- **6-этапный жизненный цикл** с гейтами (требования → дизайн → код → тесты → релиз → бой); см. `CLAUDE.md`.
- **Статистика в markdown** по принципу «летопись + сводка» (`docs/analytics/`), сводку считает детерминированный скрипт.
- **Движок качества:** `CODE-REVIEW.md` (чек-лист) + `docs/reviews/review-format.md` (формат вердикта).
- **Петля самоулучшения:** инцидент → `docs/audits/` → правило в `CODE-REVIEW.md`.

## Сознательные решения
- **Без авто-генерации знаний через git-хуки** — тихий дрейф, стоимость, не source of truth. Ручное ведение в том же PR.
- **Без QMD/семантического поиска** пока wiki <50 файлов — grep справится.
- **Аналитика в markdown, не БД/SaaS** — приватно, git-рядом, без вендор-лока.
- **Сводку считает скрипт, не LLM** — детерминированно, без дрейфа.

## Последствия
- Обновление `wiki/` — часть Definition of Done каждого PR, меняющего код.
- На выходе с каждого этапа — строка в `docs/analytics/runs.md`.
- Перед push — пересборка `dashboard.md` скриптом.
